## Аналіз

### Які функції/класи покриті найкраще?
1.Найкраще покриті функції, що входили до **Варіанту 4**:
prepareLessonSubCardCell (у файлі prepareLessonCell.js) — покрита на 100%. Я протестувала різні типи занять, обробку онлайн/офлайн локацій, а також негативні сценарії (відсутність поля room). Для досягнення максимальної надійності тестів було замокано бібліотеку i18next та застосовано строгі перевірки точних рядків (toBe) замість перевірки типів.

2.sortRooms та sortGroups — покриті на 100%. Тести включають вставку в порожній масив, вставку на початок, вставку після конкретного ID та обробку неіснуючих ідентифікаторів. Для гарантованого знищення мутацій у логічних умовах (if(afterId)) було додано спеціальні тести з масивами, що містять falsy значення (id: null).

### Які потребують додаткових тестів?
Додаткових тестів потребують глобальні утиліти та хелпери, які не входили в межі 4-го варіанту, зокрема:
- `renderScheduleTable.js` та `renderTeacherTable.js` — мають велику кількість логіки відображення, яка зараз майже не покрита.
- Саги (`src/sagas`) та сервіси (`src/services`) — вони відповідають за взаємодію з API та мають низький відсоток покриття.

### Чому деякі branches не покриті?
Низький загальний відсоток покриття гілок (Branches) пояснюється двома причинами:
1. **Специфіка завдання:** Основна увага приділялася лише трьом конкретним файлам варіанту. Величезна кількість розгалужень у файлах на кшталт `renderScheduleTable.js` залишилася поза межами тестування.
2. **Захисне програмування:** У багатьох функціях проєкту використовуються перевірки на `null` або `undefined` (Optional Chaining), які не спрацьовують під час звичайних тестів, якщо не створювати специфічні об'єкти-пустушки для кожного випадку.
